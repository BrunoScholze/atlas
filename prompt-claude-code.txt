## CONTEXTO DO PROJETO (leia depois de já ter rodado os superpowers acima)

Este é um plugin Chrome ("Agente de Chamados") que abre uma extensão do browser,
coleta dados de um ticket Jira aberto na aba ativa, e envia para um servidor Node local
que roda um agente Claude Code para analisar código e retornar onde está o bug e como
corrigir.

Arquivos do plugin (extensão Chrome):
- popup.html — interface do usuário
- popup.css  — estilos
- popup.js   — lógica do frontend (polling, renderização do resultado, etc.)

Arquivos do servidor Node (pasta raiz ou /server):
- server.js (ou index.js) — servidor Express que recebe o FormData e chama o Claude Code
- CLAUDE.md — prompt de sistema do agente
- Funcionalidades.md — mapa de funcionalidades com arquivos de cada uma

O fluxo hoje é:
1. Dev abre um ticket no Jira no browser
2. Abre o popup do plugin
3. O plugin lê automaticamente os dados do ticket da página (content script)
4. Dev pode anexar um PDF opcional e escrever uma descrição do problema
5. Clica em "Iniciar" — o servidor processa e o plugin faz polling até ter resultado
6. O resultado é renderizado com seções, diffs visuais, etc.

## PARTE 2 — FEATURE DE SELEÇÃO DE PROJETO

### Visão geral

Hoje o plugin só atende o projeto "Minha Totvs Prod". No futuro haverá outros projetos,
cada um com seu próprio conjunto de funcionalidades e codebase diferente.

Precisamos que o dev possa escolher o projeto antes de iniciar uma análise. A seleção
fica na tela principal do popup (telaFormulario), acima da seção de Ticket.

No servidor, o projeto selecionado vai determinar qual CLAUDE.md usar e quais entradas
do Funcionalidades.md são relevantes para aquele projeto.

**Implemente conforme a arquitetura que você decidiu na seção "DECISÃO DE ARQUITETURA"
acima.** Os requisitos abaixo descrevem o comportamento esperado — a estrutura de arquivos
e a forma de organizar os dados é sua decisão.

### Estrutura de dados nova

#### Arquivo: PROJETOS.md

Crie este arquivo na raiz do servidor (onde fica CLAUDE.md). Ele lista os projetos
disponíveis e seus metadados. Formato:

```
# PROJETOS

## app-minha-prod
Nome: APP Minha Prod
Descrição: Aplicativo mobile Ionic/Angular — Minha Totvs Prod
CLAUDE: CLAUDE.md
Status: ativo
```

Cada projeto tem:
- Um slug (usado internamente, ex: `app-minha-prod`)
- Nome de exibição (ex: `APP Minha Prod`)
- Descrição curta
- Qual CLAUDE.md ele usa (campo `CLAUDE:`) — permite ter um CLAUDE.md por projeto no futuro
- Status: ativo / inativo

Por enquanto só existe o projeto `app-minha-prod`. Deixe a estrutura preparada para
receber mais projetos facilmente — basta adicionar um novo bloco `## slug` no arquivo.

#### Mudança no Funcionalidades.md

Cada entrada de funcionalidade precisa indicar a qual projeto pertence.
Adicione uma tag `[projeto: app-minha-prod]` no final da linha do nome da funcionalidade,
ou no cabeçalho do bloco de cada funcionalidade — você decide o formato mais limpo,
mas deve ser parseável: o servidor vai filtrar o Funcionalidades.md pelo projeto ativo
antes de passar para o agente.

Exemplo de como pode ficar (você pode ajustar o formato, mas mantenha consistência):

```
## Apontamento de Produção [app-minha-prod]
- src/app/...
- src/app/...

## Refugo [app-minha-prod]
- src/app/...
```

Aplique a tag `[app-minha-prod]` em TODAS as funcionalidades existentes hoje no
Funcionalidades.md, já que todas pertencem a esse projeto.

### Mudanças no popup (popup.html + popup.js)

#### Nova seção no telaFormulario

Adicione uma seção ACIMA da seção de Ticket, com o seguinte comportamento:

- Label: "PROJETO" (mesmo estilo dos outros labels de seção)
- Um elemento de seleção que lista os projetos disponíveis
- Visual: não use um <select> nativo. Use um botão/pill clicável que abre um dropdown
  simples (div posicionada absolutamente) com as opções. Estilo dark, sem cores,
  sem bordas chamativas — consistente com o restante do redesign
- Quando um projeto é selecionado, o pill mostra o nome do projeto
- O projeto selecionado é enviado no FormData como campo `projeto` (o slug, ex: `app-minha-prod`)
- O primeiro projeto da lista deve ser pré-selecionado automaticamente
- O servidor retorna a lista de projetos num endpoint novo (ver abaixo); o popup
  carrega essa lista ao inicializar, exatamente como já carrega os dados do ticket

#### Endpoint novo no servidor: GET /projetos

O servidor deve ler PROJETOS.md, parsear os projetos com status "ativo" e retornar:

```json
{
  "projetos": [
    { "slug": "app-minha-prod", "nome": "APP Minha Prod", "descricao": "..." }
  ]
}
```

O popup chama esse endpoint no DOMContentLoaded, popula o dropdown e pré-seleciona
o primeiro. Se o endpoint falhar (servidor offline), o dropdown mostra "Nenhum projeto
disponível" e desabilita o botão Iniciar.

#### Envio no FormData

No `analisar()` do popup.js, inclua:
```js
formData.append('projeto', projetoSelecionado); // slug, ex: 'app-minha-prod'
```

### Mudanças no servidor Node (server.js ou index.js)

#### 1. Novo endpoint GET /projetos

Lê e parseia PROJETOS.md. Retorna apenas projetos com `Status: ativo`.

#### 2. Endpoint POST /analisar — receber e usar o campo `projeto`

Quando o campo `projeto` chegar no FormData:

a) Ler PROJETOS.md para descobrir qual CLAUDE.md usar para esse projeto
   (campo `CLAUDE:` do projeto selecionado). Se não encontrar, usar CLAUDE.md como fallback.

b) Ler Funcionalidades.md e filtrar apenas as funcionalidades que têm a tag do projeto
   selecionado (ex: `[app-minha-prod]`). Passar esse conteúdo filtrado para o agente
   em vez do arquivo completo.

   Como passar: o servidor já monta o prompt/contexto que vai para o Claude Code.
   Onde hoje passa o Funcionalidades.md completo (ou o path dele), passe apenas
   as linhas/blocos filtrados para o projeto.

c) Se o campo `projeto` vier vazio ou inválido, usar comportamento atual como fallback
   (Funcionalidades.md completo, CLAUDE.md padrão).

#### 3. Não quebrar nada existente

O campo `projeto` é novo e opcional do ponto de vista do servidor. Se não vier,
o servidor funciona exatamente como hoje. Isso garante retrocompatibilidade se
houver versão antiga do popup instalada.

### Mudança no CLAUDE.md

Adicione no bloco "O que você vai receber" o novo campo:

```
PROJETO        : <slug do projeto, ex: app-minha-prod>
```

E uma linha no bloco "Contexto do sistema":

```
- Projeto ativo: informado no campo PROJETO do input
- Funcionalidades disponíveis: apenas as do projeto ativo (já filtradas pelo servidor)
```

Não mude nenhuma outra regra do CLAUDE.md — só adicione essas informações.

### Comportamento do campo "Descrição do problema"

#### Estado atual (o que existe hoje)
O campo `descricaoTextarea` fica sempre visível, com uma dica em amarelo acima dele
dizendo que quanto mais detalhe melhor, e ele só se torna obrigatório (marcado com *)
quando não há PDF anexado.

#### Novo comportamento (o que deve ser implementado)

**O campo deve ficar oculto por padrão.** O dev só o vê se quiser adicionar contexto
extra. Há duas formas de revelar o campo:

---

**Forma 1 — Checkbox "Adicionar descrição"**

Logo abaixo da dropzone de PDF, adicione uma linha com um checkbox:

```
[ ] Adicionar descrição do problema
```

- Quando desmarcado (padrão): o textarea fica com `display: none`
- Quando marcado: o textarea aparece com uma transição suave (height de 0 para altura natural,
  ou simplesmente display block com fade-in via opacity transition 0.2s)
- O label do checkbox deve ter o mesmo estilo visual dos outros textos secundários:
  font-size 12px, cor #888888
- O checkbox em si: estilo nativo, mas dentro de um wrapper que combine com o tema dark.
  Se quiser customizar, use um checkbox estilizado com CSS (aparência de quadrado com borda
  #2E2E2E, quando marcado fundo #F0F0F0 com checkmark #0A0A0A). Não use bibliotecas externas.
- Quando o checkbox é desmarcado depois de ter sido marcado, limpar o valor do textarea
  e voltar a escondê-lo

---

**Forma 2 — Tentativa de envio sem PDF e sem descrição**

Se o dev clicar em "Iniciar" sem ter anexado PDF e sem ter texto no textarea
(mesmo que ele esteja oculto), o comportamento deve ser:

1. NÃO iniciar a análise
2. Marcar o checkbox automaticamente (revelando o textarea)
3. Aplicar um outline de atenção no textarea: borda #888888 sólida, 1px
4. Exibir uma mensagem inline abaixo do textarea (não um alert do browser):
   ```
   Anexe um PDF ou descreva o problema antes de continuar.
   ```
   Essa mensagem deve ter: font-size 11px, cor #888888, margin-top 6px
   Ela some assim que o usuário começa a digitar no textarea ou anexa um PDF
5. Dar scroll suave até o textarea para o dev não precisar procurar

Remova toda a lógica atual de `verificarBotao()` que desabilita o botão Iniciar
enquanto não há PDF nem descrição — o botão Iniciar deve ficar sempre habilitado
visualmente. A validação acontece só no momento do clique, conforme descrito acima.

---

**Lógica de obrigatoriedade (substitui a atual)**

A regra de validação no momento do clique em "Iniciar" é:
- Se `state.pdfFile !== null` → pode enviar (descrição é opcional)
- Se `state.pdfFile === null` E `descricaoTextarea.value.trim() === ''` → bloquear e avisar
- Se `state.pdfFile === null` E `descricaoTextarea.value.trim() !== ''` → pode enviar

O asterisco (*) de campo obrigatório e o elemento `descricaoObrigatorio` atual
podem ser removidos — a validação inline descrita acima substitui esse padrão.

A dica atual ("Quanto mais detalhe, melhor — informe o nome da tela...") deve ser
movida para dentro do `placeholder` do textarea, em vez de ser um bloco separado.
Isso simplifica o visual e mantém a instrução acessível:

```
placeholder="Quanto mais detalhe, melhor. Informe o nome da tela com problema e descreva o que está acontecendo — ex: Na tela de Apontamento de Produção, ao salvar, o campo Quantidade fica em branco."
```
